# decision-tree-selector

Storybook entry page for the tree-based selector runtime.

- **Hosted Storybook:** https://gitaiqaq.github.io/decision-tree-selector/
- **Repository:** https://github.com/GitaiQAQ/decision-tree-selector

`decision-tree-selector` is a React runtime for tree-based selection flows. It is built for interfaces where one choice changes which branches are visible, disabled, or selected by default.

Use this page as the reading order for the docs site: start with runtime behavior, then drill into DSL authoring, then inspect plugin-specific behavior.

## What it gives you

- a DSL for describing selector trees
- runtime bootstrap and default-selection behavior
- React providers and hooks for selection state
- a minimal built-in renderer surface
- debug snapshots for understanding runtime output

## API layers

The public surface is easiest to learn from top to bottom rather than alphabetically from the export list.

### Layer 1 — default entry point

Use `dsl`, `SelectorProvider`, and `SelectorTree` for most integrations. This is the main path demonstrated in **[Runtime/Real-World Trees](?path=/story/runtime-real-world-trees--ecommerce-catalog-tree)**.

### Layer 2 — app integration and customization

Add hooks such as `useSelectable`, `useNode`, `useCurrentRadio`, and renderer customization when your app needs controlled state, custom visuals, or richer interactions.

### Layer 3 — advanced runtime behavior

Reach for plugin helpers, `useSwitchable`, bootstrap helpers, and debug/tracking APIs when you need branch switching, policy-driven defaults, runtime snapshots, or low-level introspection.

## Where to use which entry point

- **Storybook** — best for learning the API surface and comparing runtime behaviors side by side.
- **Demo app** — best for a single bundled flow and raw runtime snapshot inspection.
- **GitHub Pages Storybook build** — best for sharing a static docs site once the Pages workflow is enabled in the repository.

## Current repository status

The docs structure is now split so DSL atoms and DSL compositions live as a first-class DSL section instead of being mixed into runtime pages.
